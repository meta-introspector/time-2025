# Idea of 1: The Atomic Unit (Scalar ExprObject)

## Concept
A single, indivisible piece of data from the log. This could be a string, number, boolean, or null. It represents the most fundamental unit in our log analysis system.

## Representation
Initially, this will be represented by a variant in our `ExprObject` enum: `ExprObject::Scalar(serde_json::Value)`. However, to meet the requirements of "location in the original text" and "tagging memory," this representation will need to evolve.

## Properties
To fully realize the "Idea of 1," a scalar `ExprObject` should encapsulate the following properties:

1.  **Value**: The actual data (e.g., "hello", 123, true, null).
2.  **Location/Offset**: Its byte range (start and end byte indices) within the original raw log string. This is crucial for:
    *   "Overlaying on memory": Allowing higher-level layers to refer directly to the original data without copying.
    *   "Each field in json is the a location in the original text": Providing precise provenance for each scalar value.
3.  **Type**: The intrinsic JSON type (string, number, boolean, null). This is already implicitly handled by `serde_json::Value`.
4.  **Tags/Constraints**: Additional metadata associated with this scalar value. This relates to the user's requirement that "other constraints can tag this memory as well." Examples could include:
    *   `sensitive_data`: Boolean flag.
    *   `PII`: Boolean flag.
    *   `error_code`: Specific error identifier.
    *   `data_quality_score`: A numerical score.

## How this relates to `zos`

The `zos` array `[0,1,2,3,5,7,11,13,17,19]` defines fundamental building blocks and their combinations. For the atomic unit:

*   `zos[0]` (0): Could represent an empty or null scalar value.
*   `zos[1]` (1): Could represent a simple, non-empty scalar value.

This suggests that even at the scalar level, the `zos` principle might influence how we categorize or tag these atomic units based on their inherent complexity or presence.

## Related Project Files

*   **`src/main.rs`**: The main entry point of the `log_analyzer` application. It represents the single point of initiation and origin for the program's execution, embodying the atomic unit of execution.

## Next Steps for Implementation

1.  **Modify `ExprObject::Scalar`**: Update its definition in `src/models.rs` to store the `serde_json::Value`, its byte offset, and its length in the original raw JSON string. This will likely involve creating a new struct (e.g., `ScalarValue`) to hold these properties.
2.  **Update `value_to_expr_object`**: Adjust the logic in `src/layers/parsing.rs` to receive and store the offset and length information when creating `ExprObject::Scalar`.
3.  **Investigate JSON parsing with byte offsets**: This remains the most challenging aspect. We need a mechanism to obtain these byte offsets during the JSON parsing process. This might involve:
    *   Exploring existing Rust JSON parsing libraries for offset capabilities.
    *   Considering a custom `serde::Deserializer` implementation.
    *   Developing a post-parsing traversal strategy to calculate offsets.
