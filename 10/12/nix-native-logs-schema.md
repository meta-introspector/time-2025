# Nix Native Logs Schema (develop2.json)

This document outlines the inferred schema for native Nix JSON logs generated by commands like `nix develop -vvvvv --log-format internal-json --print-build-logs`. The output, typically redirected to a file like `develop2.json`, contains a stream of JSON objects, each representing a log event.

**Note:** This schema is inferred based on common Nix log patterns. Actual `develop2.json` content might vary slightly. It is recommended to inspect a sample `develop2.json` for precise details.

## Overall Structure

The `develop2.json` file is expected to be a sequence of JSON objects, where each object represents a single log event. It is not a single JSON array, but rather a series of concatenated JSON objects. Each object typically has a `type` field indicating the kind of event.

## Common Log Event Types and Their Schemas

### 1. `type: "StartActivity"`

Represents the beginning of an activity.

```json
{
  "type": "StartActivity",
  "id": "<activity_id>",
  "parent": "<parent_activity_id>",
  "level": "<log_level>",
  "text": "<description_of_activity>",
  "fields": {
    "<field_name>": "<field_value>"
  }
}
```

**Fields:**
- `type` (string): Always "StartActivity".
- `id` (string): Unique identifier for this activity.
- `parent` (string, optional): ID of the parent activity, if nested.
- `level` (string): Log level (e.g., "Info", "Debug", "Verbose").
- `text` (string): Human-readable description of the activity.
- `fields` (object, optional): Additional key-value pairs providing context for the activity.

### 2. `type: "StopActivity"`

Represents the end of an activity.

```json
{
  "type": "StopActivity",
  "id": "<activity_id>",
  "status": "<activity_status>",
  "fields": {
    "<field_name>": "<field_value>"
  }
}
```

**Fields:**
- `type` (string): Always "StopActivity".
- `id` (string): ID of the activity that is stopping.
- `status` (string): Status of the activity (e.g., "Succeeded", "Failed", "Cancelled").
- `fields` (object, optional): Additional key-value pairs.

### 3. `type: "ActivityResult"`

Represents a result or output from an activity.

```json
{
  "type": "ActivityResult",
  "id": "<activity_id>",
  "level": "<log_level>",
  "text": "<result_description>",
  "fields": {
    "<field_name>": "<field_value>"
  }
}
```

**Fields:**
- `type` (string): Always "ActivityResult".
- `id` (string): ID of the activity this result belongs to.
- `level` (string): Log level.
- `text` (string): Description of the result.
- `fields` (object, optional): Additional key-value pairs, often containing specific data like build paths, error messages, etc.

### 4. `type: "Message"`

Represents a general log message.

```json
{
  "type": "Message",
  "level": "<log_level>",
  "text": "<log_message>",
  "fields": {
    "<field_name>": "<field_value>"
  }
}
```

**Fields:**
- `type` (string): Always "Message".
- `level` (string): Log level.
- `text` (string): The actual log message content.
- `fields` (object, optional): Additional key-value pairs.

### 5. `type: "BuildLogLine"`

Represents a single line from a build log (when `--print-build-logs` is used).

```json
{
  "type": "BuildLogLine",
  "id": "<activity_id>",
  "line": "<build_log_line_content>"
}
```

**Fields:**
- `type` (string): Always "BuildLogLine".
- `id` (string): ID of the build activity this log line belongs to.
- `line` (string): The actual line from the build output.

## Key Considerations for Parsing

- **Concatenated JSON:** The file is a stream of JSON objects, not a single array. Parsers should read line by line or use a streaming JSON parser.
- **Dynamic Fields:** The `fields` object within events can contain various keys depending on the specific event and Nix version. Robust parsers should be able to handle unknown fields.
- **Error Handling:** Logs can contain malformed JSON or unexpected event types. Parsers should be resilient to such issues.
- **Contextual Information:** The `id` and `parent` fields are crucial for reconstructing the hierarchy of activities and understanding the context of each log event.

This document serves as a starting point for understanding and parsing Nix native JSON logs. For precise parsing, always refer to the official Nix source code or analyze a representative `develop2.json` file.
