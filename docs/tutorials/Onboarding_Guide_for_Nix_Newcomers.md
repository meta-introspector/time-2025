# Onboarding Guide for Nix Newcomers

Welcome to the `streamofrandom` project! This guide is designed to help you get up to speed with Nix, our primary build and development environment tool, and understand its role in our project.

## 1. Introduction to Nix

Nix is a powerful package manager and system configuration tool that enables reproducible, declarative, and reliable builds.

*   **Reproducibility:** Nix ensures that building a project today will yield the exact same result tomorrow, regardless of changes in your system environment.
*   **Declarative:** You define your development environment and dependencies in a declarative way (using Nix expressions), and Nix takes care of building it.
*   **Reliability:** Nix avoids "dependency hell" by isolating packages from each other, preventing conflicts.

## 2. Nix in the `streamofrandom` Project

In `streamofrandom`, Nix is central to our development workflow, especially through the use of **Nix flakes** and our ongoing **Nixification efforts (CRQ-016)**.

*   **Nix Flakes:** Flakes are the modern way to manage Nix projects. They provide a standardized structure for defining inputs (dependencies) and outputs (packages, development environments). Our `flake.nix` file at the root of the project defines our entire development environment and all project components.
*   **Submodules and Nixification:** Many parts of our project are organized into Git submodules, each with its own `flake.nix`. Our CRQ-016 initiative focuses on standardizing these flakes and ensuring all submodules are properly Nixified, contributing to the overall purity and reproducibility of the project.

## 3. Basic Nix Commands

Here are some essential Nix commands you'll use frequently:

*   **`nix develop`**: Enters a development shell defined by `flake.nix`. This is your go-to command for setting up your environment.
*   **`nix build .#<output-name>`**: Builds a specific output (e.g., a package, an application) defined in `flake.nix`. For example, `nix build .#hello` might build a `hello` package.
*   **`nix flake update`**: Updates all inputs (dependencies) in your `flake.lock` file to their latest versions.
*   **`nix flake check`**: Checks the flake for common issues and ensures it can be evaluated.
*   **`nix run .#<app-name>`**: Runs an application defined in `flake.nix`.

## 4. Nixification Workflow

If you're contributing to our Nixification efforts or adding new components, here's a general workflow:

1.  **Understand `flake.nix` and `flake.lock`**:
    *   `flake.nix`: Defines the project's inputs (dependencies) and outputs (packages, devShells, apps).
    *   `flake.lock`: A generated file that pins the exact versions of all inputs, ensuring reproducibility. **Do not edit this file manually.**
2.  **Working with Submodules**:
    *   Each Nixified submodule will have its own `flake.nix`.
    *   When making changes within a submodule, you'll often work within its `nix develop` environment.
    *   Ensure your submodule changes are committed and pushed before updating the main project's `flake.lock` if the submodule is an input.
3.  **Testing Nix Changes**:
    *   After modifying `flake.nix` or any Nix expressions, always run `nix flake check` to catch syntax errors and basic issues.
    *   Use `nix build` to verify that your packages or applications build successfully.
    *   Run any project-specific tests within your `nix develop` environment.
4.  **Pre-commit Hooks**:
    *   Our project uses pre-commit hooks to enforce code quality and Nix best practices.
    *   Ensure you have `pre-commit install` run in your repository.
    *   These hooks will automatically run checks on your Nix files before you commit, helping you catch issues early.
